# SwiftStaticAnalysis 0.2.0 — Consolidated Audit

**Date:** 2026-05-15
**Commit at audit head:** `19c8db4` (30 commits ahead of `origin/main`)
**Scope:** full repository (`Package.swift`, ~35 KLOC across 116 Swift files in `Sources/`, plus `Tests/`, `docs/`, `.github/workflows/`, schema and DocC bundles)
**Method:** six parallel deep-dives — architecture, Swift quality, security, performance & state-of-the-art, native API alignment (via apple-docs MCP), and feature-gap (advertised vs implemented). Each deep-dive persisted under `audit/_scratch/0{1..6}-*.md` and is the source of truth for citations; this document is the synthesis.

The working tree was clean at the start of the audit, so the audited state is fully reflected in commit `19c8db4`. No code was modified during this audit.

---

## Executive summary

SwiftStaticAnalysis is an unusually mature pre-1.0 project. It has a real module graph, real perf engineering (SoA token columns, true Mersenne-61 SIMD MinHash, direction-optimising BFS, ~`Copyable` arena, bounded LRU caches), real security hardening (process env allowlist, MCP sandbox with shared `validateForReadFile`, `O_NOFOLLOW` + extension allowlist on mmap, ReDoS-aware regex on `search_symbols`), Swift 6 strict concurrency throughout, typed throws in 14+ signatures, no force-unwraps in `Sources/`, and a comprehensive test suite using Swift Testing. **The base is good.**

What 0.2.0 is missing is not engineering: it is *truth-up*. The single most important failure mode found across all six audits is a **documentation–implementation mismatch** that runs through README, CHANGELOG, SECURITY.md, and DocC. The README/CHANGELOG promises (a) SA-IS is arena-backed; (b) MinHash signatures use `InlineArray<128, UInt64>`; (c) `--parallel-mode maximum` engages AsyncChannel backpressure end-to-end; (d) `@unchecked Sendable` is dropped from the memory layer; (e) `swa unused --auto-build` exists as a CLI flag; (f) `ParallelBFS` auto-selects on graphs ≥ 1000 nodes; (g) `swa-bench` is gated in CI. **None of these is true on the default code path of `main`.** The capabilities mostly *exist as scaffolding* — the wiring into the user-visible default path is what's missing. This is also why several README "Programmatic API" snippets reference symbols (`SymbolQuery.qualifiedName(_:_:)`, `SymbolQuery.selector(_:labels:)`) that don't compile.

The rest is a roughly balanced spread: one inverted module dependency (`SymbolLookup` → `UnusedCodeDetector` purely to borrow `IndexStoreReader`), one structurally over-public layer (157 `public` declarations in `UnusedCodeDetector/DataFlow/` with zero external callers), two divergent glob matchers (`contains` vs `wholeMatch`) that produce different exclusion results within a single MCP call, a handful of MCP path-validation gaps (`index_store_path` not sandbox-checked; `ReadResource` URI parser doesn't percent-decode), and a non-trivial number of perf-on-paper wins (`Span` accessors exist but no hot path reads them; AsyncChannel exists but isn't reached by `.maximum`; bit-vector dataflow would beat the current `Set<…>` worklist by ~10–100×).

**Overall grade:** B+ on engineering, C+ on alignment between claims and reality. The fixes are concrete, mostly small, and almost entirely non-breaking.

**Risk posture:** LOW-to-MEDIUM. No critical security findings. Four HIGH-severity security items (`index_store_path` sandbox bypass, `ReadResource` URI parsing, unbounded `contextCache`, ReDoS-unguarded glob compilation reachable from MCP). Auto-grade is good because the recent v0.2 hardening (commits `8cd3541`, `741a822`, the "regex wedge" fix) is real and substantive.

## Top 12 cross-cutting findings (severity-ranked)

Each item is severity, source files, what's wrong, why it matters, and the smallest fix. Detailed citations live in `_scratch/`.

### P0 — Security & correctness (fix in 0.2.1)

#### 1. MCP `index_store_path` argument bypasses the sandbox and can write outside it
`Sources/SwiftStaticAnalysisMCP/SWAMCPServer.swift:583-586` reads `index_store_path` straight off the MCP wire into `config.indexStorePath`. `IndexStoreReader.init` (`Sources/UnusedCodeDetector/IndexStore/IndexStoreReader.swift:249`) then calls `createDirectory(at: …/IndexDatabase, withIntermediateDirectories: true)`. A hostile MCP prompt can therefore (a) make the server open `IndexStoreDB` against any directory it can read and (b) **create** an `IndexDatabase/` subdirectory at any writable location reachable by the process (e.g. `~/.ssh/IndexDatabase/`, `~/Library/LaunchAgents/IndexDatabase/`). Fix: route through `context.validatePath` at the tool handler edge. Defence-in-depth: `IndexStoreReader.init` should refuse to create `IndexDatabase/` outside a caller-supplied sandbox root, and "reader" should not perform `createDirectory` implicitly.
*(03-security.md HIGH-1.)*

#### 2. `ReadResource` URI parser is naive — latent traversal once the validator changes
`Sources/SwiftStaticAnalysisMCP/SWAMCPServer.swift:1101-1116` does `String(uri.dropFirst(7))` on a `file://…` URI with no percent-decoding, no host-component handling. A URI of the form `file://example.com/etc/passwd` silently collapses to `example.com/etc/passwd`. Today the sandbox prefix check still holds; the moment anyone refactors `validatePath` to decode `%xx` early or swaps in `URL(string:)?.path`, this becomes a sandbox escape. Fix: use `URL(string: uri)`, require `scheme == "file"` and empty/nil `host`, take `parsed.path`.
*(03-security.md HIGH-2.)*

#### 3. `SWAMCPServer.contextCache` is unbounded — DoS in long-running sessions
The parsing cache is correctly an LRU bounded to 256 (`SwiftFileParser`), but the per-codebase `contextCache` in `SWAMCPServer` is not. A hostile assistant can flood the server with unique `codebase_path` values, pinning unbounded memory. Fix: wrap in the same `OrderedDictionary`-backed LRU helper already used elsewhere in Core.
*(03-security.md HIGH-3.)*

#### 4. `CodebaseContext.matchesGlobPattern` is reachable from MCP with no ReDoS guard
The four-layer ReDoS defence (`query length cap, static antipattern prefilter, name length cap, 2.5 s budget`) is applied only in `handleSearchSymbols`. `list_swift_files.exclude_patterns` and `detect_unused_code.exclude_paths` compile arbitrary attacker-controlled regex with no length cap, no antipattern prefilter, and no wall-clock budget. Fix: lift the static antipattern check and length cap into a single `safeCompileRegex(_:)` helper and call it from every `Regex(...)` construction reachable from MCP arguments.
*(03-security.md HIGH-4.)*

#### 5. Two glob matchers with incompatible semantics produce different exclusion results in a single tool call
`UnusedCodeDetector/Filters/UnusedCodeFilter.swift:114` uses `path.contains(regex)` (partial match) and `SwiftStaticAnalysisMCP/CodebaseContext.swift:153` uses `path.wholeMatch(of: regex)` (anchored). Within `SWAMCPServer.handleDetectUnusedCode`, `CodebaseContext.findSwiftFiles` collects with `wholeMatch` and then `UnusedCodeFilter.matchesGlobPattern` post-filters with `contains`. The same `.swa.json` `excludePaths` entry can therefore behave differently depending on which stage reaches it. Fix: consolidate to a single `GlobMatcher` in `SwiftStaticAnalysisCore` with anchored `wholeMatch` semantics.
*(01-architecture.md F-02; 03-security.md HIGH-4 corollary.)*

#### 6. CLI `--parallel` maps to `.maximum` while README/`.swa.json`/`ParallelMode.from(legacyParallel:)` say `.safe`
`Sources/swa/SWA.swift:207-208,369-370` does `else if parallel { .maximum }`. `ParallelMode.from(legacyParallel:)` in Core correctly maps `true → .safe`. The CLI is the outlier and users following the README get the high-throughput preset instead of the conservative one. Fix: replace the inline branches with `ParallelMode.from(legacyParallel: parallel)`. Add `CLITests` regression.
*(01-architecture.md F-07; 06-feature-gap.md item 8.)*

### P1 — Truth-up the claims (fix before any 1.0 announcement)

#### 7. The clone-type ↔ algorithm table is misleading
README and DocC `SwiftStaticAnalysis.md:41-42` say Exact→SA-IS, Near→MinHash+LSH, Semantic→AST. The *defaults* (`--algorithm rollingHash`) dispatch Exact and Near both to `ExactCloneDetector`/`NearCloneDetector` (Rabin-Karp). MinHash+LSH only fires with explicit `--algorithm minHashLSH`, and the groups it produces are labelled `.semantic`. Either (a) flip the default to `suffixArray` for exact and `minHashLSH` for near and ensure the labels match, or (b) rewrite the table to clearly show "default vs alternative algorithms per type".
*(06-feature-gap.md headline 1.)*

#### 8. `--auto-build` flag is referenced in SECURITY.md but doesn't exist
SECURITY.md, CHANGELOG, and DocC `Security.md` repeatedly reference `swa unused --auto-build` and document the security model around it. The library field `UnusedCodeConfiguration.autoBuild` exists, the preset `.indexStoreAutoBuild` exists, but there is no CLI flag. Either add `@Flag --auto-build` to `Unused` and wire it, or rewrite the security narrative to say "auto-build is reachable only via the programmatic API / config preset" and explicitly note that the MCP tool surface intentionally does not expose it (which is correctly enforced today).
*(06-feature-gap.md C-1.)*

#### 9. README "Programmatic API" snippets don't compile
`README.md:252` and `:256` use `SymbolQuery.qualifiedName("APIClient","shared")` and `SymbolQuery.selector("fetch", labels:[…])`. Real factories: `name(_:)`, `qualified(_:)` (single dotted string), `usr(_:)`, `definition(of:)`. Either deliver the convenience initialisers or rewrite the snippets.
*(06-feature-gap.md C-2.)*

#### 10. `ParallelMode.maximum` does not engage AsyncChannel backpressure
README ("**AsyncChannel backpressure**: `ParallelMode.maximum` uses `BackpressuredChannelStream` for streaming verifiers"), CHANGELOG ("Streaming default"), and the DocC Performance article all say `.maximum` is the backpressure surface. `ParallelMode.swift:17-21` explicitly contradicts: `.safe` and `.maximum` differ only in `ConcurrencyConfiguration` shape; the streaming path is opt-in via `StreamingVerifier` / `streamResultsBackpressured`. Either branch on `.maximum` to engage the streaming path end-to-end, or soften the claim.
*(06-feature-gap.md C-3.)*

#### 11. `.swa.json` `excludeNamePatterns` / `ignoredPatterns` are decoded but never reach the detectors
Schema (`.swa.schema.json:121, 185`) advertises them. `ConfigurationLoader.swift:237, 258` merges them. `Sources/swa/SWA.swift` never reads them out into the detector configs. Silent no-op. Fix: forward through `buildUnusedConfig` / `buildDuplicationConfig` into `UnusedCodeConfiguration.ignoredPatterns` and `DuplicationConfiguration.ignoredPatterns`. Add regression tests covering the config-merge path.
*(01-architecture.md F-19; 06-feature-gap.md C-4.)*

#### 12. The README "Performance" section contains five concrete falsifiable claims that aren't true on default paths

## Per-area headlines

Each section links to the dedicated audit file under `audit/_scratch/` for full evidence.

### Architecture & module design — `_scratch/01-architecture.md` (508 lines)

Structurally solid. Three concerns worth fixing before 1.0:

- **`SymbolLookup` → `UnusedCodeDetector` is inverted.** The only thing borrowed is `IndexStoreReader`. Move it (with `IndexStoreFallbackManager`, `IndexStoreError`) to `SwiftStaticAnalysisCore/IndexStore/` or a new `IndexStoreSupport` target. Both feature modules then depend on Core. Non-breaking at the umbrella `@_exported` level. *(F-01)*
- **CFG / dataflow types are entirely `public` despite being internal infrastructure** (`UnusedCodeDetector/DataFlow/CFGBuilder.swift:27,68,88,121,153,209,300`). 157 `public` declarations create an unintentional ABI commitment. Demote to `internal`. *(F-03)*
- **`@_exported import SwiftSyntax` / `@_exported import SwiftParser` in Core, and `@_exported import MCP` in the MCP module** silently grant consumers a third-party-package surface they didn't declare. Scope to types that actually appear in this module's public signatures. *(F-04, F-11)*

Other items: README pins to `0.1.4`; `SWAMCPServer` advertises `0.1.0` despite the 0.2.0 release; `swa-mcp` parses its own argv instead of using `ArgumentParser`; `swa/SWA.swift:1059-1082` defines a `canonicalizePath` that diverges from `PathUtilities.canonicalize`; iOS 18 is declared as a platform but `IndexStoreDB` is macOS-only (and there's no iOS CI gate); `StaticAnalysisBuildPlugin` uses `.prebuildCommand` with no input tracking, so it runs on every build; `StaticAnalysisCommandPlugin` invokes `Process()` directly with the inherited environment because the SPM plugin sandbox forbids importing `ProcessExecutor` (this is documented in SECURITY.md, but the documented limitation should also have a minimal inline mitigation: `process.environment = [:]`).

### Swift quality — `_scratch/02-swift-quality.md` (400 lines)

Top-line concerns:
- **`ParallelMinHashGenerator.computeSignatures` and `ParallelProcessor.map/compactMap/forEach` ignore their own `maxConcurrency` caps** — unbounded TaskGroup vs. chunk-and-drain barriers. The streaming-bounded pattern at `ZeroCopyParser.swift:437-467` is the correct template; apply it.
- **`SoATokenStorage.append(length: Int)` silently truncates to `UInt16.max`** (`SoATokenStorage.swift:155-170`). For files with long string literals or generated multi-MB lines this produces incorrect token data with no warning.
- **`RegexCache` "LRU" eviction uses `Dictionary.keys.first`** (`RegexCache.swift:52-56`). That's undefined order, not LRU. Replace with `OrderedDictionary` (already in `Collections`).
- **`BackpressuredChannelStream.makeChannel` leaks an unstructured `Task`** (`TaskBackedAsyncStream.swift:80-89`) — the docstring promises cancellation that isn't wired up.

Strong points (verified, give credit):
- Zero `try!`, `as!`, force-unwraps in `Sources/`.
- Complete migration off `DispatchQueue`/`NSLock` to `Synchronization.Mutex`/`Atomic<UInt64>`.
- Typed throws used across 14+ APIs (Configuration, MemoryMappedFile, IndexStore, QueryParser).
- `RawSpan` accessor and `~Copyable Arena` are correctly written.
- `SyntaxArena` lifetime hygiene is right in visitors.

Native-API opportunities (these are the largest wins):
- **`Bitmap` (298 LOC) duplicates `Collections.BitArray`** — replace with the package implementation.
- **`import Foundation` everywhere; `FoundationEssentials` is never used.** On macOS 15, ~88 of 90+ files can switch and remove the implicit ObjC runtime from library targets.
- **String paths everywhere; `URL.FilePath` / `System.FilePath` never used.** `swift-system` is already transitively resolved (1.6.3).
- **`ObjCBool` smuggled via `FileManager.fileExists(atPath:isDirectory:)` in 4 places.**
- **`RawSpan(_unsafeBytes:)` uses an underscored, non-source-stable initialiser.**
- **Zero parameterised tests** despite Swift Testing being used throughout; tags defined in 5 of 90+ suites.

### Security — `_scratch/03-security.md` (942 lines)

Risk rating **LOW-to-MEDIUM**. No critical findings. Four HIGH (covered in §"Top 12" above as P0-1..4). Selected MEDIUM/LOW items:

- `read_file` extension allowlist includes `.plist`, `.json`, `.yaml`. For projects with committed credential files (`fastlane/Appfile`, `.env.json`, etc.) this is a data-exfiltration vector under MCP.
- `MemoryMappedFile` is SIGBUS-vulnerable to truncation under a memory-mapped fd. CLI doesn't care; MCP does — a hostile codebase root could truncate a file the server already mapped.
- `getCodebaseInfo` reads every Swift file with `String(contentsOfFile:)` with no per-file cap. Worst case: huge generated `.swift` files exhaust memory.
- `swift-lmdb` is pinned by branch in `Package.resolved` (upstream-imposed; mitigable via `--frozen` in CI).
- The two `Process` invocations in the SPM command plugin (`StaticAnalysisCommandPlugin/Plugin.swift:51, 123`) inherit the parent environment — known limitation per SECURITY.md, but `process.environment = [:]` adds defence at zero cost.

What the project already gets right (verified, not flattery): canonical path validation in `CodebaseContext`, shared `validateForReadFile` between `read_file` and `ReadResource`, environment allowlist in `ProcessExecutor`, `O_NOFOLLOW` + `S_IFREG` check in `MemoryMappedFile`, four-layer ReDoS defence in `search_symbols`, partial-results-with-warning UX on budget exhaustion, stdlib `Synchronization` primitives, bounded LRU parser cache, honest `auto_build` carve-out from the MCP surface, no network code anywhere, well-documented `@unchecked Sendable` discipline.

### Performance & distance from SOTA — `_scratch/04-performance.md` (877 lines)

The six claim-vs-code gaps are covered in §"Top 12" above (item 12). Beyond those, the real algorithmic concerns:

- **AST fingerprint uses `% 1_000_000_007` (a 30-bit prime).** At N ≈ 50 000 snippets the expected collision rate is ~50%. Switch to M_61 (already used correctly in MinHash) — about a 10-line change in `ASTFingerprintVisitor.swift`.
- **`ReachingDefinitions.swift:363` is O(B²)** because the worklist is a `Set<BlockID>` and ordering is restored every iteration via `cfg.reversePostOrder.first(where: worklist.contains)`. Replace with an indexed priority deque keyed on RPO index.
- **All intraprocedural dataflow uses `Set<DefinitionSite>` / `Set<VariableID>`.** Bit-vectors via the existing `Bitmap` would be 10–100× faster for the typical Swift function shape.
- **`LSHIndex.optimalBandsAndRows` uses symmetric error.** Clone detection wants recall over precision; bias the error.
- **`MultiProbeLSH` perturbations are not probability-ordered** — not actually Lv et al. 2007. Currently it's a strictly enumerated probe set, not a multi-probe distribution.
- **`AtomicBitmap`'s storage is `[AtomicWord]` (boxed atomics + array bounds-check per `testAndSet`).** A `ManagedBuffer`-backed flat contiguous store is the right shape. (Cross-referenced with native-API audit; see §native-apis Topic 1.)

Strengths called out honestly: Mersenne-61 SIMD4 reduction is real and correct; Beamer BFS uses the right "remaining edges" heuristic with documented α/β defaults; `DenseGraph` caching is correctly invalidated; env-allowlist scrubbing is solid; `~Copyable` `Arena` is the modern shape; `OrderedDictionary` LRU is O(1) per touch.

Distance-to-SOTA scorecard (full version in scratch): SA-IS roughly 3–5× slower than `libsais`; MinHash variance higher than BBit/DartMinHash; AST fingerprint less robust than SourcererCC bag-of-words+filtering; reachability competitive with Roslyn dead-code; symbol search competitive with SourceKit-LSP locally but doesn't speak SCIP/LSIF.

### Native API alignment (apple-docs MCP) — `_scratch/05-native-apis.md` (720 lines)

The 15 topics in this audit are evidence-heavy with citations to apple-docs MCP pages. Highlights:

- **`Synchronization.Mutex<State>` and `Atomic<UInt64>` are used correctly** at `MemoryMappedFile.swift:316`, `SymbolFinder.swift:46`, `Bitmap.swift:17`. The `AtomicBitmap` wrapper allocates one `final class AtomicWord` per 64-bit word — a workaround for `Atomic<T>: ~Copyable` not fitting in `Array`. A `ManagedBuffer`-backed flat store would reduce ~1024 object allocations to one for a 65 536-node graph.
- **`Span<T>` / `RawSpan` / `~Escapable`** are partially adopted. `MemoryMappedFile.withRawSpan(borrowing)` is canonical; `SoATokenStorage` doesn't expose `Span<T>` accessors yet (`kindSpan`/`offsetSpan` would make the zero-copy contract explicit). `FileSlice: @unchecked Sendable` is a candidate for `~Escapable` — its lifetime is tied to its `MemoryMappedFile`.
- **`Foundation` vs `FoundationEssentials`**: 90+ files import `Foundation`. On macOS 15, `FoundationEssentials` covers `URL`, `FileManager`, `JSONDecoder`, `JSONEncoder`, `Data`, `ProcessInfo` — the vast majority of use-sites. Only `ProcessExecutor.swift` (uses `Process`) and `PathUtilities.swift` (uses `NSString.expandingTildeInPath`) strictly need full Foundation.
- **`swift-system` / `FileDescriptor` / `FilePath`**: `MemoryMappedFile` uses raw POSIX `open`/`mmap`/`munmap`. `swift-system` is transitively resolved. Replacing `open(path, O_RDONLY|O_NOFOLLOW)` with `FileDescriptor.open(FilePath(path), .readOnly, options: .noFollow)` gives typed errors and moves a step toward removing `@unchecked Sendable` from this layer.
- **`withDiscardingTaskGroup` would replace `withThrowingTaskGroup(of: Void.self)`** at `ConcurrencyConfiguration.swift:173`. Two-line change; eliminates per-task result-slot allocation.
- **OSSignposter is entirely absent.** `swa-bench` has no signpost phase markers, making Instruments profiling opaque. `AnalysisLogger` wraps `OSLog.Logger` correctly with `privacy: .public`, but for perf-oriented work `os_signpost` is the documented Apple guidance.
- **MCP swift-sdk usage is idiomatic** (`SWAMCPServer` is an `actor`, uses `StdioTransport`). The hardcoded `version: "0.1.0"` and unbounded `contextCache` are the only real issues.
- **`swift-argument-parser` usage is correct** (`AsyncParsableCommand` throughout, `throw ExitCode(2)` for findings). `swa-mcp` is the outlier — it parses its own argv.
- **Swift 6.2 features**: typed throws used in 14 signatures, ahead of community curve. `@concurrent` and `DefaultIsolation(.mainActor)` correctly avoided for a library+CLI target.

### Feature gap (claims vs reality) — `_scratch/06-feature-gap.md` (903 lines)

Grade **C+**. Top items already covered in §"Top 12" as items 7–12. Additional notable drift:

- **Schema/default mismatch**: `.swa.schema.json` declares `treatPublicAsRoot/Objc/Tests/SwiftUIViewsAsRoot` default `false`. CLI and library `UnusedCodeConfiguration.init` default them to `true`. Update the schema.
- **`--min-confidence low` cannot relax the detector's `.medium` default** — it acts as a post-filter only; the detector won't emit below-medium findings to filter.
- **Build plugin invokes `swa analyze … --format xcode` as a `prebuildCommand`.** `Analyze.run` throws `ExitCode(2)` on findings. As a prebuild, that fails the build. README doesn't mention this. Either flip the plugin to a command plugin, or invoke a `--no-fail` variant.
- **Legacy `parallel` "deprecated" with no deprecation diagnostic emitted.** Add `Diagnostic.print("--parallel is deprecated; use --parallel-mode")`.
- **`swa analyze` accepts only `--config` and `--format`** — can't override unused/duplicates flags from the CLI without `.swa.json`. Undocumented limitation.
- **Top-level `.swa.json` `format` field is decoded but never applied** by any subcommand.
- **Library presets unreachable from CLI**: `.strict`, `.hybrid`, `.indexStoreAutoBuild`, `.incremental`, `.highPerformance`, `.type3Detection`.
- **`--report` silently ignored when `--mode` ≠ `reachability`.** Should error or note in the output.
- **`swa:ignore:begin/end` and `swa:ignore-duplicates:begin/end` range markers are parsed** by `IgnoreDirectiveScanner` but not in README.
- **`AUDIT.md` referenced by CHANGELOG line 10 is missing** from the repo (this audit fills that gap).

## Things this project already does well

These are not consolation prizes — they are the reason 0.2.0 deserves a B+ at the engineering level.

1. **Real Swift 6 strict concurrency.** Every target sets `.swiftLanguageMode(.v6) + .enableExperimentalFeature("StrictConcurrency")`. The `@unchecked Sendable` discipline is principled at every site.
2. **`Synchronization.Mutex` / `Atomic<UInt64>` end-to-end.** No leftover `NSLock`, `os_unfair_lock`, `DispatchSemaphore`, `DispatchQueue`. The 0.2.0 dropoff of `swift-atomics` is correct.
3. **Typed throws in 14+ APIs** (`ConfigurationError`, `MemoryMappedFileError`, `IndexStoreError`, `QueryParseError`) — ahead of community curve.
4. **No `try!` / `as!` / force-unwraps in `Sources/`.** Enforced by `.swift-format` (`NeverForceUnwrap`, `NeverUseForceTry`).
5. **Sandbox model in the MCP server.** `validatePath` does separator-aware prefix checking after canonical resolution. `validateForReadFile` is shared between two callers — the right way to avoid security-control drift.
6. **`ProcessExecutor` environment allowlist** is conservative and explicit. `DYLD_INSERT_LIBRARIES`, `DEVELOPER_DIR`, `SWIFTPM_HOOKS_DIR`, `BASH_ENV` are all closed.
7. **`MemoryMappedFile`** uses `O_NOFOLLOW`, refuses non-regular files via `S_IFREG`, caps at 256 MiB, and exposes `withRawSpan(borrowing)`.
8. **Mersenne-61 SIMD MinHash** is genuine: branchless reduction, `SIMD4<UInt64>` lanes for `(a,b,sig)`, four hash universes per shingle, SplitMix64 mixing.
9. **`DenseGraph` + Beamer-style direction-optimising BFS** is sound, with explicit α/β thresholds. Cache invalidation is correct.
10. **Module cycle resolution is real.** The `Bitmap`/`AtomicBitmap` move from `UnusedCodeDetector` to Core in 0.2.0 cleanly severed the prior reverse dependency.
11. **Umbrella split (`SwiftStaticAnalysis` vs `SwiftStaticAnalysisAll`)** is correct — analyzer-only consumers don't pull in `modelcontextprotocol/swift-sdk`.
12. **Configuration validation at the load boundary** with typed `throws(ConfigurationError)` and structured error messages.
13. **Exit-code contract is correctly wired** (0/2/64/1). The README claim is honoured.
14. **`~Copyable` `Arena`** and `ThreadLocalArena` are the modern shape.
15. **`SyntaxArena` lifetime hygiene** in visitors is correct — no captured nodes outliving their parse trees.
16. **Comprehensive test suite** using Swift Testing (`@Test`, `#expect`). Tags exist (use them more).
17. **Honest carve-out**: the MCP tool surface deliberately does not expose `auto_build`. The documentation around this is correct (even if SECURITY.md confusingly implies a non-existent `--auto-build` CLI flag).

## A note on the user's question about "what the tool announces vs what it actually fulfils"

The blunt summary is: SwiftStaticAnalysis 0.2.0 fulfils more than half of what it announces, but the gap is structural rather than scattered. The 12 highest-impact items in this audit are all *truth-up* problems — either deliver the wired-up version of a capability that exists as scaffolding (auto-build flag, ParallelMode.maximum → backpressure, arena-backed SA-IS, InlineArray signatures, `excludeNamePatterns` plumbing) or honestly soften the claim. Three are actual security gaps (MCP `index_store_path`, `ReadResource` URI, glob ReDoS) and two are bugs (`--parallel` flag mapping, `SoATokenStorage` silent truncation).

The good news: every item in the P0 list is a small, surgical change that can land in 0.2.1, and the result will be a project whose advertised surface matches its implementation. That is what a 1.0 needs.
